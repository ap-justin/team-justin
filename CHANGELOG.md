## v0.23.0 — `typescript` shared skill: TypeScript as ambient craft, not a seat
The team's framework builders each *write* TypeScript, but nothing captured the shared discipline underneath them: a sane tsconfig baseline (strictness, module resolution, aliases, project references, the typecheck gate) and the hard type work (gnarly generics, inference, branded types, `.d.ts`, cryptic errors). First cut was a standalone `typescript-expert` **seat** — then we caught that TypeScript is unlike the other specialist seats. `better-auth`/`ark-ui`/`postgres`/`sanity` each own a **bounded external surface** handed off as a discrete artifact; TypeScript is **ambient** — woven through every line a builder writes, with no separable "the types" artifact to hand back. A seat would only add a fuzzy builder/expert seam + an extra opus hop for routine typing. So it's a **skill every builder carries**, not a seat.
- **New `skills/typescript/SKILL.md`** (repo-authored, model-invocable) — the playbook: consult-docs-first, the **cheat-sheet baseline** (Matt Pocock's `tsconfig-cheat-sheet` — base opts, strictness trio, `module preserve` for bundlers / `nodenext` for `tsc`, DOM-vs-Node `lib`, library `declaration`/`composite`), tsconfig discipline (extend framework presets, never fork; verify via `tsc --showConfig`), type-level programming (generics/conditional/mapped/branded types, discriminated unions, `.d.ts`, `declare module`, reading cryptic errors), ESM/CJS, monorepo project references (`tsc -b`), JS→TS migration, and type-perf (`--generateTrace`).
- **Loaded by the 8 TS-writing builders** — `sveltekit`/`react-router`/`nextjs`/`sanity`-builder, `better-auth-specialist`, `ark-ui-specialist`, `postgres-architect`, `test-writer` each got a short `## TypeScript (shared skill)` pointer (same wiring pattern as the v0.19.0 context-hygiene block). Builders solve type work in-context — no handoff, no seam.
- **Repo-wide TS-infra work stays dispatchable** without a standing seat: the lead sends a general agent with the `typescript` skill for a strict migration / monorepo references / type-perf profiling. Mint a dedicated seat only if that recurs (same logic as the planned `astro-builder`).
- **Excludes the formatter/linter** (Biome/ESLint/Prettier) — a separate tooling concern, deliberately set aside (a biome-only seat can follow).
- Wiring: `skills/typescript/` (new); 8 `agents/*.md` (skill pointer); `skills/engineering-team` (routing row removed + "TypeScript isn't routed" note); `ROSTER.md` (header 0.22.0 → **0.23.0**, shared-skill line under Reused-not-owned; no specialist/model-tier row); `SOURCES.md` (TypeScript row reframed to the skill); `VERSION`. Minor bump — new capability. *(The earlier `typescript-expert` seat draft never shipped — superseded by the skill in the same version.)*

## v0.22.0 — color as a scale of solid steps (no opacity as a lightener)
The design layer's color contract was 4–6 loose hexes (`design-director`: "roles: bg, surface, text, accent"). That underspecifies every intermediate shade — so builders reach for opacity to fake them (`bg-accent/10`, `rgba(brand,.1)`), which makes contrast non-deterministic over varying backgrounds and reads as templated. Adopted the **principle** behind Radix's 12-step scale (not its palette): every color is an ordered set of **solid** steps, each step with a fixed job. Anti-slop *and* an a11y win.
- **`agents/design-director.md`** — Color output reworked from "4–6 named hex values" to **solid stepped scales**: a neutral + accent scale, each an ordered set of solid steps mapped to the 12-step semantic model (1–2 bg · 3–5 component bg+states · 6–8 borders · 9–10 solid · 11–12 text). Not all 12 mandatory, but every shade the UI needs is its own solid step; dark mode is a second solid scale, not the light one dimmed. New **hard rule — no opacity as a lightener**: alpha is for genuine translucency only (glass/scrim/disabled), never to derive a lighter/darker shade.
- **`agents/taste-reviewer.md`** — new mechanical FAIL **Opacity-as-lightener**: greppable check for `rgba(…,0.NN)` / `hsl(… / .NN)` / Tailwind `bg-*/NN` `text-*/NN` `opacity-NN` / `color-mix(… white/black)` used to tint a solid color, cite utility+file:line, fix = next solid scale step. The director removes the *need*; the reviewer catches the *violation*.
- **Seams unchanged**: `design-director` authors the scale, builders consume it, `ark-ui-specialist` already "applies the director's tokens, doesn't invent them" — so it inherits the scale with no edit. Vendored `accessibility-review` left byte-identical (re-syncable).
- Wiring: `agents/design-director.md`, `agents/taste-reviewer.md`; new `SOURCES.md` row (Design color system → Radix "Understanding the scale", principle-not-palette); `ROSTER.md` header 0.21.0 → **0.22.0**; `VERSION`. Minor bump — new capability/discipline, no orchestration-contract change.

## v0.19.0 — context-hygiene floor for builders + scoped delegation
A single builder was burning 400k+ tokens on one ticket. The orchestrator can't cap a subagent mid-run (isolation protects the *lead's* context, not the child's), so the fix is upstream: scope what goes in, and give builders a hygiene floor. Two sinks addressed — builders re-exploring for their own files, and doc/MCP over-fetching + re-reads.
- **Context-hygiene block on every builder/specialist** — new `## Context hygiene (stay lean)` section in `sveltekit-builder`, `react-router-builder`, `nextjs-builder`, `sanity-builder`, `better-auth-specialist`, `postgres-architect`, `test-writer`: read only what the brief names (broad search is `Explore`'s job, not a builder's); never re-read a just-edited file; fetch the narrowest doc section and don't re-pull docs already in context (Svelte: no autofixer re-run on unchanged components; Sanity: `get_schema` once; etc.); flag when a task is too big for one run instead of sprawling.
- **Scoped delegation contract** — `engineering-team` Step 3: "Pass the FULL relevant context" → pass it **scoped** (exact files/paths/ranges from `Explore`'s map, so the builder doesn't re-discover them). Named the failure mode: a builder hunting for its own files is the #1 way a run sprawls to hundreds of K tokens.
- **Split-on-size rule** — new Rules bullet: a build that would touch many files/subsystems gets split across sequential builders (or `planner` slices), not handed to one builder. "Scope in, don't cap after." Companion Rules bullet ties the every-hop context rule to the same scoping discipline.
- Wiring: 7 `agents/*.md` (hygiene block); `skills/engineering-team` (Step 3 scoped-context line + two Rules bullets). No roster/source changes — behavior floor, not a new seat.

## v0.18.0 — reconcile the plan at commit + a zero-friction idea Icebox
The `management/` store was well-specified as a *store* but had **no reconciliation loop** — nothing tied a commit to the plan, so code could land while its ticket still said `todo` (frontier lies) or a shipped roadmap item sat in Now. And a mid-task idea (user-pitched or team-discovered) had **no lightweight home**: the only way to touch the roadmap was spawning the AFK `product-manager`, so ideas got dropped. Closed both.
- **Lead reconciles at commit** — new `engineering-team` **Step 4.5**: whenever a `management/` store exists, the lead updates it as part of the *same commit* that lands each slice. Auto (write, then report): flip every ticket whose acceptance boxes are all satisfied to `status: done` + recompute/report the frontier; move any now-complete roadmap item (Now → shipped). Plan and code share one git history — `git log` never shows code ahead of a stale plan; the reconciled state rides into the PR. **Lead-owned, no git hook** (a hook can't judge *which* ticket a commit completes).
- **The Icebox — captured, untriaged** — a trailing `## Icebox (captured, untriaged)` section in `ROADMAP.md` (not a new file, not a brief). Zero-friction, pre-triage parking lot deliberately **outside** curated Now/Next/Later so raw ideas don't pollute the prioritized horizons. The lead appends a one-liner (`<idea> — _pitched|discovered · effort: <slug> · <date>_`) the moment an idea surfaces and keeps going — never derails the task.
- **`product-manager` drains it** — triages Icebox lines on each run: promote into a real horizon item (brief + score, evidence-grounded) or drop. Draining queue, not an ever-growing dump; an un-triaged line is a reminder, never a commitment.
- Wiring: `skills/engineering-team` (Step 4.5 + Rules bullet); `TRACKER.md` (new *Reconcile & capture* section + Icebox spec, roadmap-ops rows); `agents/product-manager.md` (Icebox intake + triage in the store section); `ROSTER.md` (header **v0.15.2 → v0.18.0** — was stale through 0.16/0.17; product-manager row + plan-store line). ## v0.17.0 — ticket state in frontmatter, one file per ticket
v0.16.0 moved the plan store into git-tracked markdown but left the machine-relevant facts encoded as prose: status = "are the acceptance boxes ticked" (scrape the body), edges = **Blocked by** *titles* (string-match). Both are brittle — title edges break on rename, box-scanning breaks on formatting drift. Fixed by structuring the ~3 fields that are actually queried, leaving the prose body (the 95% that's genuine spec) as markdown.
- **One file per ticket.** `plan/<effort>/tickets.md` → one `<nnn>-<slug>.md` per ticket under `plan/<effort>/tickets/` (`<nnn>` = zero-padded dependency-order sequence, gaps left to insert). Cleaner diffs, stable ids, fewer merge conflicts editing tickets in sequence.
- **Typed YAML frontmatter** on each ticket: `id` (stable join key — edges point at it, so renames don't break), `status: todo|doing|done` (the machine roll-up; `doing` = claimed, replacing wayfinder's per-issue assignee), `blocked_by: [ids]`. Acceptance `- [ ]` boxes stay as the builder's checklist/evidence; the builder flips `status: done` once they're all checked + verified. **The frontier is now a deterministic query** — `status != done` with all `blocked_by` ids done — not a prose scan.
- **Ids join, titles narrate.** Frontmatter edges use ids; everything a human reads still refers to a ticket by title-wrapping-its-link (unchanged rule).
- **Briefs** gain an `id` field (alongside `horizon` + `status: ready-for-planning`).
- **Vendored skills stay verbatim.** This amends the `<tickets-file-template>` (single-file, prose-only) as a `TRACKER.md` local override — keep the skill's body, wrap it with frontmatter — so `to-tickets`/`wayfinder` remain re-syncable from upstream.
- Wiring: `TRACKER.md` (layout, Status & frontier, planning ops, naming); `agents/planner.md` (store + to-tickets/wayfinder modes); `agents/product-manager.md` (brief frontmatter); `SOURCES.md` planner row; `skills/engineering-team` Step 2.6 dispatch line.

## v0.12.0 — true-alpha cutouts (rembg) for graphic-designer
- **`graphic-designer` gains real background removal**: a `--cutout` path that produces a genuine 4-channel-alpha cutout (transparent subject on nothing), for portraits/logos/products that must sit flat on a page with no card or matte.

Runs **fully local, no `GOOGLE_API_KEY`**; self-bootstraps a dedicated python venv (`scripts/.venv-rembg`, gitignored) with the first python (3.12→3.11→3) that has onnxruntime wheels; the onnx model caches to `~/.u2net`. Default model `u2net_human_seg` (people); `--rembg-model u2net` for objects/products. Webp output now sets `alphaQuality: 100` so cutout edges stay crisp.
- **Doc/routing fix (the real lesson)**: `agents/graphic-designer.md` previously listed "background removal" under the gemini `--input` edit — the stale claim that sent the lead down the wrong path. Split it: gemini `--input` = opaque relight/restyle only; `--cutout` = true-alpha. Frontmatter `description` now names cutouts so the at-a-glance routing signal matches reality (same class of fix as v0.11.1).
